In questo caso installeremo OTRS 3 anche se ancora in beta. Questo perchè è ritenuto abbastanza stabile (attualmente siamo già alla terza beta) e ci evita di dover effettuare un upgrade da una versione 2.x ad una versione 3.x quando la 3 diventerà stabile (è rinomato che passare da una versione ad un altra, per qualsiasi programma, non è una cosa facile).
Scaricare dal sito l'archivio compresso: http://otrs.org/download/ (la registrazione non è obbligatoria, in fondo alla pagina c'è l'opzione Skip this step).
UPDATE 2013.11.14: a quel link non riesco più a trovare l'archivio per l'installazione. Andare quindi su https://www.otrs.com/try/ e scaricare l'ultima versione di OTRS Help Desk. A questo indirizzo sono comunque presenti tutte le versioni: ftp://ftp.otrs.org/pub/otrs/
Scompattarlo in /opt (o in una directory a piacere).
Installazione delle dipendenze necessarie.
installare apache con mod perl e mysql-server; su qualsiasi debian based:
# aptitude install apache2 libapache2-mod-perl2 mysql-server

installare i moduli perl necessari:
a questo scopo, otrs mette a disposizione uno script che indica quali moduli sono installati e quali no: /opt/otrs/bin/otrs.CheckModules.pl (basta lanciarlo per ottenere l'elenco). L'elenco segna alcuni moduli come opzionali, noi cerchiamo di installarli tutti per completezza; l'unico modulo che non riesco a trovare è Encode::HanExtra ma, oltre ad essere opzionale, la descrizione (Required to handle mails with several Chinese character sets) ci fa capire che a noi non serve. Con le distribuzioni debian based, i moduli perl sono presenti come normali pacchetti e il loro nome finisce per *-perl. Un altro modo per cercare i pacchetti corrispondenti ai moduli perl, è utilizzare apt-cache, es. apt-cache search GD::Text (dove GD::Text è il nome del modulo perl che ci interessa). Nel mio caso installo tutti i moduli mancanti con il comando:

aptitude install  libgd-text-perl libgd-graph-perl libjson-xs-perl \
libio-socket-ssl-perl libnet-dns-perl libnet-imap-simple-ssl-perl \
libnet-smtp-ssl-perl libnet-ldap-server-perl libpdf-api2-perl \
libsoap-lite-perl libtext-csv-xs-perl libxml-parser-perl libmail-imapclient-perl

Se apache era già installato, riavviarlo per fargli trovare il mod_perl:

# /etc/init.d/apache2 restart

Creare un utente apposito per otrs sulla macchina

# useradd -d /opt/otrs/ -c 'OTRS user' otrs

Aggiungere l'utente al gruppo del webserver

# usermod -G www-data otrs

Copia dei file di configurazione; nell'archivio i file di configurazione hanno nome *-dist e vanno quindi rinominati:

# cp /opt/otrs/Kernel/Config.pm.dist /opt/otrs/Kernel/Config.pm
# cd /opt/otrs/Kernel/Config
# for foo in *.dist; do cp $foo `basename $foo .dist`; done

NOTE: a me risulta che in quella dir ci sia solo un file *.dist, non so perchè faccia il for, l'ho copiato dall'INSTALL
Controllo che i moduli necessari siano installati:

# perl -cw /opt/otrs/bin/cgi-bin/index.pl
# perl -cw /opt/otrs/bin/otrs.PostMaster.pl

entrambi questi comandi devono restituire: syntax OK
Impostazioni webserver; occorre impostare apache per interpretare la cartella di otrs con perl, creare il file /etc/apache2/conf.d/otrs con il seguente contenuto

# agent, admin and customer frontend
ScriptAlias /otrs/ "/opt/otrs/bin/cgi-bin/"
Alias /otrs-web/ "/opt/otrs/var/httpd/htdocs/"
 
# load all otrs modules
Perlrequire /opt/otrs/scripts/apache2-perl-startup.pl
 
# Apache::Reload - Reload Perl Modules when Changed on Disk
PerlModule Apache2::Reload
PerlInitHandler Apache2::Reload
PerlModule Apache2::RequestRec
 
# set mod_perl2 options
<Location /otrs>
    #ErrorDocument 403 /otrs/customer.pl
    ErrorDocument 403 /otrs/index.pl
    SetHandler  perl-script
    PerlResponseHandler ModPerl::Registry
    Options +ExecCGI
    PerlOptions +ParseHeaders
    PerlOptions +SetupEnv
    Order allow,deny
    Allow from all
</Location>

Riavviare apache in modo che legga le modifiche al file di configurazione.

Settare i permessi sui file; otrs mette a disposizione uno script che si occupa di settare i permessi passando come parametri l'utente otrs di sistema, il gruppo di tale utente (solitamente uguale al nome utente), l'utente che esegue il webserver e il gruppo del webserver, nel nostro caso:

# /opt/otrs/bin/otrs.SetPermissions.pl --otrs-user=otrs \
--web-user=www-data --otrs-group=otrs --web-group=www-data /opt/otrs

Completamento dell'installazione di otrs dall'interfaccia web http://yourhost/otrs/installer.pl:
dopo aver accettato la licenza, compare la form per i dati del database. Nei primi campi vanno inseriti i dati amministrativi di mysql (root). Cliccando il pulsante “Check database settings”, il programma controlla che i dati siano corretti, e mostra un'altra parte di form. Questa nuova parte permette di specificare i dati dell'utente mysql che otrs utilizzerà per gestire i database (e avrà permessi limitati solo su di esso) e il nome del database. I campi sono già popolati con valori di default che a noi vanno bene.

ATTENZIONE: si consiglia di cambiare la password di default (hot) in qualcosa d'altro.

Continuando, l'installer crea utente e database coi dati specificati nella form.
Continuando si arriva ad un'altra form che permette di impostare le ultime cose:
SystemID: va bene quello di default (10)
FQDN: Full qualified domain name (per ora, test, l'ip?)
AdminEmail: la propria email
Organization: il nome dell'azienda (YetOpen S.r.l.)
LogModule: va bene quello di default (syslog)
LogFile: va bene quello di default (/tmp/otrs.log)
Default language: Italiano
CheckRecordMX: Yes
Impostazioni email:
skip

 

Per loggarsi la prima volta i dati saranno
user: root@localhost
pass: root

Il sistema ci avvisa subito con un messaggio che è bene non lavorare con utenti amministrativi e che la password di default del programma non è sicura. Bisogna pertanto cambiare la password e creare nuovi utenti per l'utilizzo dell'applicazione.

 

I cronjobs di otrs sono nella directory var/cron nella directory home dell'utente otrs. Per prima cosa occorre rinominare tutti i file *.dist senza l'estensione (cron ignora infatti i file di configurazione che contengono il carattere '.') dopodiché occorre eseguire lo script Cron.sh start per caricare il cron dell'utente. Di seguito i comandi per eseguire le operazioni appena descritte:

# cd /opt/otrs/var/cron
# for foo in *.dist; do cp $foo `basename $foo .dist`; done
# chown root:www-data *
# su - otrs
$ bin/Cron.sh start


fonte: http://doc.otrs.org/3.2/en/html/manual-installation-of-otrs.html#cronjobs
