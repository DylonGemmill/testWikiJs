Le macchine di gestione del DNS di MilanoTeleport sono due Arch Linux su cui gira il server DNS Bind. 

Il server dns1 è il master, dns2 lo slave. Le modifiche fatte sul primo sono propagate automaticamente sul secondo.

Il demone viene eseguito in modalità chroot, il che significa che le configurazioni vengono prese da /etc/bind/chroot/etc/bind (e non /etc/bind).

Attenzione: c'è però uno script che gira giornalmente che copia le configurazioni presenti in /etc/bind in /etc/bind/chroot/etc/bind! Quindi le modifiche vanno fatte in entrambi i percorsi: il primo per la persistenza ed il secondo per provarle.

In data 2016.11.10 abbiamo però sospeso l'esecuzione dello script schedulato, perché riavviando bind azzerava le statistiche tutte le notti. Quindi per ora le configurazioni attive sono solo quelle della chroot. Dettagli sullo script in fondo alla pagina.

Per avviare, fermare e riavviare il demone si usa systemctl, per cui i comandi da lanciare sono rispettivamente:

systemctl start bind-chroot
systemctl stop bind-chroot
systemctl restart bind-chroot

oppure ancora status per vederne lo stato.

 

Se non specificato diversamente tutti i percorsi sono relativi alla directory di configurazione indicata sopra.

Aggiunta di un record PTR ad una zona esistente

L'elenco delle zone inverse (ptr) configurate è visibile con il comando:

ls zones/master/reverse/

I file di zona sono specificati al contrario, con il suffisso in-addr.arpa. Il contenuto di una zona, ad esempio 111.29.46.in-addr.arpa,  è il seguente:

; Revision History: April 22, 1999 - admin@mail.openna.com
; Zone file for 111.29.46.in-addr.arpa
; (c) MilanoTeleport 2009
; Start of Authority (SOA) records.
$TTL 86400
@  IN  SOA  dns1.milanoteleport.com. netnoc.milanoteleport.com.  (
        2016110702      ; Serial
        3600   ; Refresh
        900    ; Retry
        1209600 ; Expire
        86400 ); Minimum

; Name Server (NS) records.
@       IN      NS   dns1.milanoteleport.com.
@       IN      NS   dns2.milanoteleport.com.

; only PTR records.

53      IN    PTR  mailserver.mv-oceanatlantic.com.
186     IN    PTR  mailserver.mv-victory.com.
236     IN    PTR  mailserver.mv-seaadventurer.com.

In questo caso sono definiti tre record PTR per 46.29.111.53, .186 e .236. 

Per aggiungere un nuovo record ci sono due cose da fare: aggiungere il record e modificare il seriale del file di zona.
Apriamo il file con un editor (ad esempio nano, o vim), aggiungiamo una riga in fondo copiando il formato dalle esistente (quindi il primo campo rappresenta l'ultimo ottetto dell'IP, IN e PTR rimangono invariate, ed in fondo va messo l'hostname. Attenzione: è importante che l'hostname sia seguito dal punto).
La modifica del numero di serie serve a far sapere al server slave che c'è stato un aggiornamento della zona. Se non viene modificato il seriale non avverrà alcun aggiornamento. Di norma si consiglia di usare come seriale la data odierna più un progressivo. Nell'esempio sopra la zona è stata modificata due volte in data 2016/11/07.

Fatte le modifiche al file basterà riavviare il servizio per applicare le modifiche. Come detto sopra quindi:

systemctl restart bind-chroot

Non dovrebbe apparire nessun messaggio e comparirà immediatamente sotto la shell. Per verificare che il servizio sia attivo:

systemctl status bind-chroot

e dovrebbe comparire la riga con Active: active (running) in verde.

Non è necessario riavviare Bind sul server slave perché la configurazione viene replicata automaticamente.

Aggiunta di una nuova zona di risoluzione inversa (PTR)

In caso sia necessario aggiungere una nuova zona:

  1. modificare il file reverse.conf, copiando il blocco di una zona modificando la descrizione della zona e il nome del file;
  2. copiare il file di una zona esistente presente in zones/master/reverse/ con il nuovo nome specificato in reverse.conf, facendo attenzione a rimuovere eventuali record PTR già presenti;
  3. riavviare bind-chroot.

 

Script giornaliero di aggiornamento DNS

Lo script è schedulato sul crontab di root (crontab -l root), ed è in /opt/create_zones/dns_master_zones_update.sh. In riepilogo le operazioni che vengono eseguite sono:

  1. download da 195.120.182.203 (Ministero dell'Interno) della lista dei siti pedopornografici censurati;
  2. download dei siti bloccati da AAMS;
  3. esecuzione dello script /opt/create_zones/create_master_zones.py per convertire i siti scaricati in zone per bind;
  4. copia dei file di config di bind da /etc/bind in /etc/bind/chroot/etc;
  5. riavvio bind;
  6. invio email a tommaso.pappagallo@milanoteleport.com, customercare@milanoteleport.com con le modifiche effettuate al DNS.

I server DNS erano probabilmente originariamente concepiti per essere usati dai clienti di MT, per la risoluzione normale. Poi sono stati configurati anche per le zone inverse.
