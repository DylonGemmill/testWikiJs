Requisiti

Il testing è basato su PHPUnit, installabile tramite pacchetto Ubuntu.

apt-get install phpunit phpunit-selenium
pear install phpunit/dbUnit

 

 

Fixture

La configurazione base di un'applicazione Yii prevede già le directory per lo unit testing. 

Come prima cosa occorre creare un database dedicato con la struttura del database di produzione. Fatto questo creiamo un file di config che punti a questo DB. Modificare il file config/test.php con qualcosa di simile:

<?php return CMap::mergeArray( require(dirname(__FILE__).'/main.php'), array( 'components'=>array( 'fixture'=>array( 'class'=>'system.test.CDbFixtureManager', ), 'db'=>array( 'connectionString' => 'mysql:host=localhost;dbname=hotspotv2servertest', 'emulatePrepare' => true, 'username' => 'hotspot', 'password' => 'hotpass9', 'charset' => 'utf8', 'enableProfiling'=>true, 'enableParamLogging'=>true, ), 'log'=>array( 'class'=>'CLogRouter', 'routes'=>array( array( 'class'=>'CFileLogRoute', 'levels'=>'error, warning, trace, debug', 'logFile'=>'test.log', ), ), ), ), ) );

 

Due cose fondamentali: il caricamento del fixture manager e l'override della definizione del database.

 

In alcuni casi è utile abilitare il debug e il logging come in fase di sviluppo. Per questo modificare bootstrap.php aggiungendo prima della prima require:

defined('YII_DEBUG') or define('YII_DEBUG',true); defined('YII_TRACE_LEVEL') or define('YII_TRACE_LEVEL', 3);

e dopo tutto:

// automatically send every new message to available log routes Yii::getLogger()->autoFlush = 1; // when sending a message to log routes, also notify them to dump the message // into the corresponding persistent storage (e.g. DB, email) Yii::getLogger()->autoDump = true;

Altrimenti lasciare invariato...

Per effettuare i test del web, ma anche delle migrazioni, dobbiamo creare degli index dedicati. La index-test.php dovrebbe già esistere che punta a config/test.php, mentre per la console creare un file yiic-test.php con questo contenuto:

#!/usr/bin/env php 
 <?php  
// change the following paths if necessary $yiic=dirname(__FILE__).'/../../yii/framework/yiic.php'; 
$config=dirname(__FILE__).'/config/console-test.php';  
// Debug 
defined('YII_DEBUG') or define('YII_DEBUG',true); 
 require_once($yiic);

 

Per i test web dobbiamo specificare l'url a cui accedere in modo da usare il database di test invece di quello di produzione/sviluppo. Aprire quindi WebTestCase.php e modificare la definizione di TEST_BASE_URL. Se nell'esecuzione dei test per il web ci sono problemi con la selezione del browser è possibile forzare Firefox aggiungendo 

$this->setBrowser('*firefox');

 

prima della setBrowserUrl. 

 

Definizione dei dati

I test devono essere eseguiti su una base dati definita, questo compito è assolto dalle fixtures. Per ogni test dovremo dire quali fixture usare, e per ognua di esse Yii provvederà a svuotare la tabella ed inserire i dati presenti nelle definizioni. Nelle fixture andiamo quindi di fatto a definire delle righe di dati con dei valori da usare in seguito nei test. Il nome file della fixture deve essere il nome della tabella, deve essere salvato nella directory fixtures (sempre in tests), il suo contenuto una return array con l'assegnamento tra nome campo => valore. Ogni riga è un array, la sua chiave può essere omessa, ma in caso venga specificata può essere usata nei test per accedere velocemente alla riga specifica. E' buona norma usare una riga per un test specifico senza riciclarle, in modo da non creare problemi con i dati durante i controlli (ad esclusione delle dipendenze tra test). 
Ecco un esempio di definizione di fixture:

<?php
return array (
    'ca1'=>array(
        'id'=>1,
        'codice'=>'ca01',
        'descrizione'=>'Catena test 01',
    ),
    'ca2'=>array(
        'id'=>2,
        'codice'=>'ca02',
        'descrizione'=>'Catena test 02',
    ),
);

In questo caso ci sono due righe, ca1 e ca2. Più tardi nel test potremo usare $this->nomefixture('ca1) per caricare l'ActiveRecord relativo a quella riga. 

 

Scrivere i test per il database

I test veri e propri vanno nella directory unit, e sono salvati in dei file con il nome del modello e la desinenza Test, quindi ad esempio CatenaTest.php per il modello Catena. Per ogni test posso definire delle fixtures, quindi le tabelle che andranno azzerate prima dell'esecuzione, e una serie di metodi che sono le azioni da eseguire. Tipicamente il nome del test si rifà al metodo della classe che andremo a testare. L'esempio pratico per il modello Catena è il seguente:

<?php

class HotspotTest extends CDbTestCase
{
    public $fixtures=array(
        'catena'=>'Catena',
    );

    public function testConfig ()
    {
        $c = $this->catena('ca1');           
}

 

Per accedere alle righe delle fixture abbiamo due opzioni:

  • array: è possibile accedere ai soli dati della definizione usando 
    $this->nomefixture['ca1']['descrizione']
  • AR: è possibile caricare l'AR relativo alla riga usando
    $this->nomefixutre('ca1')->descrizione 
Per verificare effettivamente che un'azione faccia quello che ci aspettiamo dobbiamo usare le asserts. Ad esempio per vedere che il salvataggio dell'AR non dia errore:
$this->assertTrue($model->save());
Oltre ad assertTrue abbiamo un elenco lungo di metodi, ma i più usati sono assertTrue, assertNull, assertNotNull, assert[Not]Equals. Nel caso della Equals dobbiamo dare come primo paragrafo il valore aspettato, come secondo la variabile. Nel caso di prima:
$this->assertEquals(true,$model->save());

 

Test funzionali (web)

Oltre ai test del db è possibile verificare anche le funzionalità della parte web servita agli utenti. Ad esempio è possibile simulare un login, simulare la compilazione di un tasto e verificare che i dati scritti siano corretti.

Questi test sono nella directory functional, ed invece che ereditare da DbTestCase prendono da WebTestCase.

Prima di eseguire questi test dovremo avere l'applet server di Selenium attiva. Per eseguirlo scaricare il jar e prima di lanciare il test eseguire java -jar selenium.jar.

Anche in questi test dobbiamo definire le fixture da usare, dopodiché provvediamo a scrivere il test vero e proprio. Ad esempio per un test di login:

<?php

class HotspotLoginTest extends WebTestCase {
    public $fixtures=array(
        'hotspot'=>'Hotspot',
        'federazione'=>'Federazione',
        'catena'=>'Catena',
        'prepagata'=>'Prepagata',
        'prepagata_anagrafica'=>'PrepagataAnagrafica',
    );
    
    public function testLogin() {
        $hs = $this->hotspot("hs1");
        $this->open('/hotspotLogin');  
$this->assertElementPresent('name=HotspotFormLogin[username]');
        $this->assertElementPresent('name=HotspotFormLogin[password]');
        if ($hs->getConfigVal('opzioni.checkbox_ricorda_credenziali')) {
            $this->assertElementPresent('name=HotspotFormLogin[rememberme]');
        } else {
            $this->assertElementNotPresent('name=HotspotFormLogin[rememberme]');
        }
}

 

 

 

Eseguire un test

Per lanciare un test posizionarsi nella directory test ed eseguire

phpunit unit/CatenaTest.php

E' anche possibile eseguire un solo test all'interno del file usando l'opzione --filter=nomeTest.
