Unit testing

Gli unit test rappresentano la parte più bassa dei test, e servono a verificare che i metodi implementati restituiscano sempre i valori attesi.

Generalmente si crea un test per ogni modello, ed al suo interno test per le singole funzioni. È importante che i test siano comprensibili, quindi meglio crearli con nomi anche prolissi, in modo che si capisca bene cosa stiamo andando a testare. Ad esempio testDiLoginConUtenteValido, testDiLoginOmettendoIlCampoPassword e così via. Tutti i metodi che implementano un test devono iniziare con test.

È molto importante effettuare anche dei test negativi, non solamente quelli positivi: ad esempio verificare che passando un parametro non compatibile la funzione torni un'eccezione.

I test si basano sulle Assertions, ovvero dichiarazioni di quello che ci si aspetta. Tipicamente si usa assertTrue, oppure assertEquals e così via. Ogni chiamata accetta sempre un ultimo parametro che è il messaggio: è molto utile quindi va sempre compilato. Se scriviamo un messaggio dettagliato e significativo sarà più semplice per una persona esterna capire l'errore e cosa ci si aspetta.

Per generare un test

vendor/bin/codecept generate:test unit LezioneTest

Verrà creato il file tests/unit/LezioneTest.php che eredita da \Codeception\Test\Unit.

 

    public function testFindUserById()
    {
        $user = Utente::findIdentity(1);
        $this->assertNotNull($user, "L'utente con ID 1 dovrebbe esistere");
        $this->assertEquals($user->username, 'support@yetopen.it', "L'indirizzo email dell'utente ID 1 dovrebbe essere support@yetopen.it");

        $this->assertNull(Utente::findIdentity(999), "Non si dovrebbe trovare l'utente con ID 999");
    }

È possibile eseguire più volte lo stesso test con valori diversi usando i data provider: definiamo un array di valori e li passiamo alla funzione. Ad esempio nella funzione sopra se separiamo il test negativo per l'utente 999 e lo facciamo su diversi valori, come ad esempio -1, null, 999, modifichiamo la funzione per ricevere un parametro e dichiariamo il dataprovider nei phpdoc:

    /**
     * @dataProvider idUtentiNonEsistenti
     */
    public function testRicercaUtenteNonEsistente($id)
    {
        $this->assertNull(Utente::findIdentity($id), "Non si dovrebbe trovare l'utente con ID <{$id}>");
    }

    public function idUtentiNonEsistenti()
    {
        return [[-1], [999], [null]];
    }

 

Fixture

Le fixture sono un modo un po' più organizzato e consistente dei dataprovider sopra: sono dei dati confezionati appositamente per i test, che consentono di avere informazioni predeterminate da testare.

 

Faker è un componente che consente di generare delle fixture da dati casuali secondo delle regole. L'implementazione di Yii2 è yii2-faker, e per utilizzarla (da testyii solamente) modificare config/test_console.php aggiungendo:

    'controllerMap' => [
        'fixture' => [
            'class' => yii\faker\FixtureController::class,
        ],
    ],

 

Gitlab Runner

È il sistema che gestisce le build di prova e compilazione. Viene installato su un server (tipicamente diverso da Gitlab, ma con Docker anche sulla stessa macchina), ma può essere installato anche sui client per testare il file di build o effettuare le attività pianificate.

Per installare vedere la guida.

Installazione su Linux

sudo wget -O /usr/local/bin/gitlab-runner https://gitlab-runner-downloads.s3.amazonaws.com/latest/binaries/gitlab-runner-linux-amd64
sudo chmod +x /usr/local/bin/gitlab-runner
curl -sSL https://get.docker.com/ | sh
sudo useradd --comment 'GitLab Runner' --create-home gitlab-runner --shell /bin/bash
sudo gitlab-runner install --user=gitlab-runner --working-directory=/home/gitlab-runner
sudo gitlab-runner start

Effettuare ora la registrazione del runner su GL con questi comandi:

sudo gitlab-runner register

I parametri:

  • Installazione: https://code.ufficyo.com
  • Token: token di registrazione presente nell'amministrazione > Overview > Runners
  • Executor: docker

Per gli altri parametri vanno bene le informazioni predefinite.

 

Installazione su macOS

sudo curl --output /usr/local/bin/gitlab-runner https://gitlab-runner-downloads.s3.amazonaws.com/latest/binaries/gitlab-runner-darwin-amd64
sudo chmod +x /usr/local/bin/gitlab-runner

La registrazione avviene allo stesso modo della controparte Linux, con la differenza che il comando va lanciato senza sudo:

gitlab-runner register

Dopo la registrazione è possibile installare il servizio ed eseguirlo. L'installazione si può omettere se non si vuole riavviare automaticamente dopo il boot:

gitlab-runner install
gitlab-runner start
